# 产品经理技能

## 概述

本技能提供专业的产品管理能力,包括产品设计、需求分析、文档编写等功能。帮助将模糊的需求转化为清晰的产品方案和规范的产品文档。

## 核心能力

### 1. 需求分析与产品规划

- 收集和分析用户需求,识别核心痛点和机会
- 进行竞品分析,了解市场定位和差异化
- 制定产品愿景和战略目标
- 设计产品路线图,明确优先级和时间规划

### 2. 产品文档编写

#### 产品需求文档(PRD)
编写完整的产品需求文档,包含以下章节:
- **文档概述**: 项目背景、目标、范围
- **用户画像**: 目标用户群体、使用场景
- **功能需求**: 详细的功能描述、优先级
- **非功能需求**: 性能、安全、可用性要求
- **用户体验**: 交互流程、界面设计要点
- **技术要求**: 技术栈、依赖、集成需求
- **验收标准**: 功能测试、性能指标

#### 用户故事(User Stories)
按照标准格式编写用户故事:
- **作为** [角色]
- **我想要** [功能/需求]
- **以便** [价值/目标]

每条用户故事应包含:
- 详细的验收标准(Acceptance Criteria)
- 故事点估算
- 依赖关系说明

### 3. 产品设计

#### 功能设计
- 功能流程图和状态机设计
- 信息架构设计
- 数据模型设计

#### 交互设计
- 用户旅程图(User Journey Map)
- 原型线框图描述
- 交互规范说明

### 4. 产品管理最佳实践

#### 需求优先级排序
使用以下方法进行优先级决策:
- **MoSCoW法则**: Must have, Should have, Could have, Won't have
- **Kano模型**: 基本型、期望型、兴奋型需求
- **RICE评分**: Reach(触达) × Impact(影响) / Effort(工作量) × Confidence(信心)

#### 迭代规划
- Sprint planning 和版本迭代
- 发布计划制定
- 风险评估和应对策略

## 文档编写规范

### 文档结构
```
# [文档标题]

## 1. 文档信息
- **版本**: v1.0
- **创建日期**: YYYY-MM-DD
- **作者**: [作者姓名]
- **状态**: [草稿/评审中/已发布/已废弃]

## 2. 目录
[自动生成目录]

## 3. 正文内容
[按章节组织内容]

## 4. 附录
- 相关文档链接
- 参考资料
```

### 编写原则
- **清晰性**: 使用简洁明了的语言,避免歧义
- **完整性**: 覆盖所有关键方面,不遗漏重要信息
- **可读性**: 使用标题、列表、表格等结构化表达
- **可追溯性**: 需求项应有唯一标识,便于跟踪
- **可验证性**: 每个需求都应有明确的验收标准

## 工作流程

### 产品设计流程
1. **需求收集** → 收集用户反馈、市场调研、业务需求
2. **需求分析** → 分析需求合理性、优先级、依赖关系
3. **方案设计** → 设计功能方案、交互方案、技术方案
4. **文档编写** → 编写PRD、用户故事、设计文档
5. **评审验证** → 与团队评审、收集反馈、迭代优化

### 文档评审要点
- 需求是否清晰明确
- 优先级是否合理
- 验收标准是否可执行
- 是否考虑边界情况和异常处理
- 技术实现是否可行

## 质量检查清单

在交付产品文档前,确认以下项目:

### PRD质量检查
- [ ] 项目背景和目标是否清晰
- [ ] 用户画像和使用场景是否准确
- [ ] 功能需求是否完整无遗漏
- [ ] 非功能需求是否考虑充分
- [ ] 验收标准是否可测试
- [ ] 优先级是否合理
- [ ] 是否考虑了技术可行性

### 用户故事质量检查
- [ ] 故事是否符合"作为...我想要...以便..."格式
- [ ] 验收标准是否明确可测试
- [ ] 故事大小是否合理(建议2-5天内完成)
- [ ] 是否有明确的依赖关系
- [ ] 是否可独立交付价值

## 常见场景处理

### 需求冲突处理
当遇到需求冲突时:
1. 识别冲突的本质(优先级、资源、技术限制)
2. 分析各方案的成本和收益
3. 寻求折中方案或替代方案
4. 与相关方沟通确认
5. 记录决策依据

### 需求变更管理
- 评估变更的影响范围
- 评估变更的成本和风险
- 与团队和干系人沟通
- 更新相关文档和计划
- 保持变更记录可追溯

## 使用说明

### 何时使用本技能
- 用户提到产品设计、功能设计
- 需要编写产品文档(PRD、用户故事等)
- 进行需求分析和规划
- 创建产品路线图
- 进行产品评审和优化

### 如何有效使用
1. 明确产品目标和用户群体
2. 提供充分的需求背景和上下文
3. 说明期望的交付物格式和详细程度
4. 保持沟通,及时反馈和调整

## 5. SOP引擎产品参考

### 5.1 产品定位

**核心价值**:
- 自动化SOP(标准作业程序)提取和管理
- 智能化对话合规性验证
- 提升销售质量,降低合规风险

**目标用户**:
- 销售管理人员: 管理SOP模板,监控团队表现
- 销售人员: 获取话术指导,自我提升
- 合规人员: 审查对话记录,确保合规

### 5.2 核心功能

#### 功能1: SOP模板提取

**用户故事**:
```
作为销售管理人员,
我希望上传PDF/DOCX文档,
系统能自动提取SOP模板,
这样我就不需要手工整理话术
```

**验收标准**:
- [ ] 支持PDF和DOCX格式
- [ ] 自动识别模板名称和步骤
- [ ] 提取准确率>90%
- [ ] 提取时间<5分钟/文档

#### 功能2: 对话验证

**用户故事**:
```
作为合规人员,
我希望系统自动验证销售对话,
检查是否符合SOP规范,
这样就能快速发现违规话术
```

**验收标准**:
- [ ] 验证步骤完整性
- [ ] 检测禁忌语和敏感词
- [ ] 提供违规位置和原因
- [ ] 验证时间<1分钟/对话

#### 功能3: 对话分类

**用户故事**:
```
作为销售管理人员,
希望系统自动分类对话,
统计不同产品类型的销售情况,
这样就能了解团队销售重点
```

**验收标准**:
- [ ] 支持多产品类型分类
- [ ] 分类准确率>85%
- [ ] 提供分类统计报表

### 5.3 产品指标

**业务指标**:
- SOP模板数量: 目标100+个
- 日验证对话量: 目标1000+条
- 验证准确率: 目标>90%
- 用户满意度: 目标>4.5/5

**技术指标**:
- API响应时间: P95<2秒
- 任务处理时间: 平均<30秒
- 系统可用性: >99.9%
