# 项目经理技能

## 技能概述

本技能提供专业的项目管理知识和工作流程,帮助推动项目从规划到交付的全生命周期管理。重点关注任务规划、进度跟踪、风险控制、质量保证和团队协作。

## 使用场景

当满足以下任一条件时,应使用此技能:

- 需要制定项目计划和里程碑
- 需要跟踪任务进度和项目状态
- 需要进行风险评估和风险应对
- 需要协调团队资源和协作
- 需要进行项目评审和质量检查
- 需要生成项目报告和总结
- 需要制定开发流程和规范

## 一、项目规划与里程碑管理

### 1.1 项目阶段划分

根据 LifeManager 项目特点,采用以下阶段划分:

```
Phase 0: 项目初始化
├── 需求分析
├── 技术选型
├── 团队组建
└── 环境搭建

Phase 1: MVP 开发
├── 产品设计
├── 技术架构设计
├── 后端开发
├── 前端开发
└── 测试验收

Phase 2: 功能增强 (未来)
├── 提醒通知
├── 数据统计
└── 多平台支持
```

### 1.2 里程碑定义

每个阶段应设置明确的里程碑:

| 阶段 | 里程碑 | 验收标准 | 优先级 |
|------|--------|---------|--------|
| Phase 0 | 技术选型完成 | 技术架构文档完成,技术栈确定 | P0 |
| Phase 0 | 环境搭建完成 | 开发环境可运行,CI/CD 配置完成 | P0 |
| Phase 1 | 产品设计完成 | 产品需求文档、UI设计文档完成 | P0 |
| Phase 1 | 后端开发完成 | 所有 API 接口开发完成,测试通过 | P0 |
| Phase 1 | 前端开发完成 | 所有页面开发完成,可联调 | P0 |
| Phase 1 | MVP 上线 | 核心功能可用,通过验收测试 | ⭐ 核心 |

### 1.3 里程碑验收流程

当团队认为里程碑完成时,按照以下流程验收:

```
1. 自我验证
   ├── 检查所有任务是否完成
   ├── 检查文档是否完整
   └── 检查测试是否通过

2. 提交验收
   ├── 创建 Pull Request
   ├── 填写验收清单
   └── 通知相关人员

3. 代码审查
   ├── 技术架构师审查代码质量
   ├── 产品经理审查功能完整性
   └── 测试人员测试功能

4. 验收通过
   ├── 合并代码到主分支
   ├── 更新项目状态
   └── 生成里程碑报告
```

## 二、任务规划与跟踪

### 2.1 任务分解原则

将项目分解为可管理的小任务,遵循以下原则:

- **单一职责**: 每个任务只解决一个问题
- **可估时**: 任务可在 0.5-2 天内完成
- **可验证**: 任务完成有明确的验收标准
- **依赖清晰**: 任务间的依赖关系明确

### 2.2 任务优先级定义

| 优先级 | 说明 | 完成时间 | 响应速度 |
|--------|------|---------|---------|
| P0 | 阻塞性任务,必须立即处理 | 1 天内 | 2 小时内 |
| P1 | 高优先级,影响核心功能 | 3 天内 | 4 小时内 |
| P2 | 中优先级,不影响核心功能 | 1 周内 | 24 小时内 |
| P3 | 低优先级,可延后 | 2 周内 | 48 小时内 |

### 2.3 任务跟踪方法

使用 TODO 工具跟踪任务:

```python
# 创建任务
todo_write(merge=False, todos=[
    {"id": "1", "status": "pending", "content": "任务描述"},
    {"id": "2", "status": "pending", "content": "任务描述"}
])

# 更新任务状态
todo_write(merge=True, todos=[
    {"id": "1", "status": "in_progress", "content": "任务描述"}
])

# 完成任务
todo_write(merge=True, todos=[
    {"id": "1", "status": "completed", "content": "任务描述"}
])
```

### 2.4 日常站会流程

建议每日举行 15 分钟站会,回答三个问题:

1. **昨天完成了什么?**
   - 列出已完成的任务
   - 遇到的问题和解决方案

2. **今天计划做什么?**
   - 列出今天的任务计划
   - 标注需要的协作

3. **有什么阻碍?**
   - 列出遇到的问题
   - 需要的支持和资源

## 三、风险管理与应对

### 3.1 风险识别

项目中的常见风险:

| 风险类型 | 风险描述 | 影响程度 | 发生概率 |
|---------|---------|---------|---------|
| 技术风险 | 技术选型不当,无法满足需求 | 高 | 中 |
| 进度风险 | 任务估时不准,项目延期 | 高 | 高 |
| 质量风险 | 测试不充分,Bug 较多 | 中 | 中 |
| 人员风险 | 关键人员离职或请假 | 高 | 低 |
| 需求风险 | 需求变更频繁 | 中 | 高 |

### 3.2 风险应对策略

根据风险等级,采取不同的应对策略:

#### 高影响 + 高概率 → 规避风险
- 提前进行技术验证
- 预留 20% 缓冲时间
- 加强代码审查和测试

#### 高影响 + 低概率 → 降低风险
- 准备备用方案
- 关键人员知识共享
- 定期备份代码和数据

#### 低影响 + 高概率 → 接受风险
- 建立快速响应机制
- 预先准备解决方案
- 定期监控风险指标

#### 低影响 + 低概率 → 忽略风险
- 记录风险,定期检查
- 无需主动应对

### 3.3 风险监控

定期检查风险状态,更新风险矩阵:

```
风险矩阵 (每周更新)

| 风险 | 当前状态 | 应对措施 | 负责人 |
|------|---------|---------|--------|
| AI API 限流 | 🟢 已解决 | 增加重试机制,缓存结果 | 后端开发 |
| 进度延期 | 🟡 监控中 | 调整任务优先级,增加资源 | 项目经理 |
| 需求变更 | 🔴 高风险 | 与产品经理确认,控制变更范围 | 项目经理 |
```

## 四、质量保证与代码审查

### 4.1 代码审查流程

每个 PR 必须经过代码审查:

```
1. 开发者提交 PR
   ├── 完成代码编写
   ├── 编写单元测试
   ├── 自测功能
   └── 提交 PR 并填写描述

2. 自动检查
   ├── Linter 检查
   ├── 单元测试通过
   └── 构建成功

3. 人工审查
   ├── 至少 1 人审查
   ├── 检查代码质量
   ├── 检查测试覆盖
   └── 提出修改意见

4. 修改与再审查
   ├── 开发者修改代码
   ├── 重新提交
   └── 审查通过

5. 合并
   ├── Squash and merge
   ├── 删除分支
   └── 更新文档
```

### 4.2 代码审查清单

审查代码时,检查以下项目:

#### 代码质量
- [ ] 代码符合命名规范
- [ ] 函数单一职责,逻辑清晰
- [ ] 适当使用注释
- [ ] 避免重复代码
- [ ] 异常处理完善

#### 测试覆盖
- [ ] 单元测试覆盖率 > 80%
- [ ] 边界条件测试
- [ ] 异常情况测试
- [ ] 集成测试通过

#### 文档更新
- [ ] API 文档更新
- [ ] README 更新
- [ ] 变更日志记录
- [ ] 知识库更新

### 4.3 测试标准

| 测试类型 | 覆盖率要求 | 执行时机 |
|---------|-----------|---------|
| 单元测试 | 核心逻辑 > 90%,整体 > 80% | 每次提交 |
| 集成测试 | 核心流程 100% | 合并前 |
| 端到端测试 | 核心功能 100% | 发布前 |
| 性能测试 | 响应时间符合要求 | 上线前 |

## 五、团队协作与沟通

### 5.1 协作工具使用

- **代码管理**: Git + GitHub/GitLab
- **任务管理**: TODO 列表 + GitHub Issues
- **文档管理**: Markdown 文档
- **沟通协作**: 实时讨论 + 文档评审

### 5.2 团队角色与职责

| 角色 | 主要职责 | 技能调用 |
|------|---------|---------|
| 产品经理 | 需求分析、产品设计、功能优先级 | `产品经理` |
| 技术架构师 | 技术选型、架构设计、代码审查 | `技术架构师` |
| 后端开发工程师 | API 开发、数据库设计、AI 集成 | `python后端开发工程师` |
| 前端开发工程师 | 页面开发、交互实现、联调 | `vue3前端开发工程师` |
| UI/UX 设计师 | UI 设计、交互设计、原型制作 | `ui/ux设计师` |
| DevOps 工程师 | CI/CD、部署、监控、运维 | `devops工程师` |
| 项目经理 | 项目规划、进度跟踪、风险管理 | `项目经理` |

### 5.3 沟通规范

#### 异步沟通
- 所有讨论应在文档中进行
- 重要决策必须记录在文档
- 使用 @ 提及相关人员
- 及时回复和确认

#### 同步沟通
- 站会每天 15 分钟
- 周会每周 1 小时
- 重要变更召开会议
- 会议后发送纪要

## 六、文档管理

### 6.1 文档分类

根据 LifeManager 项目,文档分类如下:

```
docs/
├── phase0/          # 项目初始化文档
│   ├── 需求分析.md
│   ├── 技术选型.md
│   └── 团队组建.md
├── phase1/          # MVP 开发文档
│   ├── 产品需求文档.md
│   ├── UI设计文档.md
│   ├── 技术架构设计.md
│   ├── 后端开发指南.md
│   ├── API接口文档.md
│   └── 开发指南文档.md
└── phase2/          # 未来功能文档 (待创建)
```

### 6.2 文档更新流程

```
1. 需求变更
   ├── 产品经理发起
   ├── 评估变更影响
   └── 更新相关文档

2. 技术决策
   ├── 技术架构师发起
   ├── 记录决策过程
   └── 更新技术文档

3. 开发过程
   ├── 开发过程中更新
   ├── 记录问题和解决方案
   └── 补充示例代码

4. 里程碑完成
   ├── 汇总变更内容
   ├── 生成里程碑报告
   └── 更新项目状态
```

### 6.3 文档质量标准

| 文档类型 | 必须包含 | 可选内容 |
|---------|---------|---------|
| 需求文档 | 功能描述、验收标准 | 用户画像、使用场景 |
| 设计文档 | 架构设计、技术选型 | 设计图、UML 图 |
| API 文档 | 接口定义、示例代码 | 错误码说明、测试用例 |
| 开发文档 | 环境配置、开发流程 | 最佳实践、调试技巧 |
| 知识库 | 核心概念、常见问题 | 详细说明、参考资料 |

## 八、敏捷开发实践

### 8.1 迭代规划

建议采用 2 周一个迭代:

```
Iteration N (2 周)
├── 第 1-2 天: 迭代规划
│   ├── 确定迭代目标
│   ├── 分解任务
│   └── 分配任务
├── 第 3-10 天: 开发执行
│   ├── 每日站会
│   ├── 代码审查
│   └── 持续集成
└── 第 11-14 天: 迭代评审
    ├── 功能演示
    ├── 迭代回顾
    └── 下一迭代规划
```
