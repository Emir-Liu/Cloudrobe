## 2. 架构概述

### 2.1 系统定位
云衣橱是一个基于P2P模式的服装租赁共享平台，采用前后端分离架构，通过HBuilderX开发的uni-app前端（支持微信小程序、H5、App）作为表现层，FastAPI作为后端，实现衣物的发布、租赁、订单管理等核心功能。

### 2.2 架构模式
采用**分层架构** + **领域驱动设计(DDD)**：
- **表现层**: HBuilderX + uni-app（微信小程序/H5/App）
- **API层**: FastAPI RESTful API
- **业务逻辑层**: 领域服务
- **数据访问层**: ORM
- **基础设施层**: 数据库、缓存、对象存储

### 4.2 分层架构设计

#### 表现层 (Presentation Layer)
- **职责**: 接收HTTP请求，参数验证，返回响应
- **技术**: FastAPI路由、Pydantic验证
- **示例**:
```python
@app.post("/api/v1/clothing")
async def create_clothing(
    request: ClothingCreateRequest,
    current_user: User = Depends(get_current_user)
):
    # 参数验证由Pydantic自动完成
    clothing = await clothing_service.create(request, current_user)
    return ClothingResponse.from_entity(clothing)
```

#### 业务逻辑层 (Service Layer)
- **职责**: 核心业务逻辑，领域服务编排
- **技术**: Python异步函数
- **示例**:
```python
class ClothingService:
    async def create(self, request: ClothingCreate, user: User):
        # 业务规则验证
        if not user.is_verified:
            raise BusinessError("请先完成实名认证")
        
        # 创建衣物
        clothing = Clothing(**request.dict(), owner_id=user.id)
        
        # 上传图片
        images = await upload_images(request.images)
        clothing.images = images
        
        # 保存
        clothing = await self.repo.save(clothing)
        
        # 发布事件
        await event_bus.publish(ClothingCreatedEvent(clothing))
        
        return clothing
```

#### 数据访问层 (Repository Layer)
- **职责**: 数据库操作抽象
- **技术**: SQLAlchemy ORM
- **示例**:
```python
class ClothingRepository:
    async def save(self, clothing: Clothing):
        async with db.session() as session:
            session.add(clothing)
            await session.commit()
            await session.refresh(clothing)
        return clothing
    
    async def find_by_id(self, clothing_id: int):
        return await db.session().get(Clothing, clothing_id)
```

#### 基础设施层 (Infrastructure Layer)
- **职责**: 外部服务集成（支付、物流、OCR等）
- **技术**: 第三方SDK
- **示例**:
```python
class WeChatPayment:
    async def pay(self, order_id: str, amount: int):
        # 调用微信支付API
        result = await wechat_pay_client.pay(order_id, amount)
        return result
```

### 4.3 领域模型设计

#### 聚合根 (Aggregate Root)
1. **User (用户)**: 管理用户信息、信用积分
2. **Clothing (衣物)**: 管理衣物信息、租赁记录
3. **Order (订单)**: 管理订单状态、租赁流程

#### 值对象 (Value Object)
1. **Money (金额)**: 租金、押金、运费
2. **Rating (评分)**: 用户评分、衣物评分
3. **DateRange (日期范围)**: 租赁日期

#### 领域事件 (Domain Event)
1. **ClothingCreatedEvent**: 衣物创建
2. **OrderCreatedEvent**: 订单创建
3. **OrderCompletedEvent**: 订单完成
4. **CreditChangedEvent**: 信用积分变更

## 7. 安全设计

### 7.1 认证授权
- **认证方式**: JWT Token
- **Token有效期**: 7天
- **Token刷新**: 自动续期
- **权限控制**: RBAC（角色权限）

### 7.2 数据加密
- **传输加密**: HTTPS/TLS
- **密码加密**: bcrypt
- **敏感数据**: AES-256加密
- **身份证号**: 加密存储 + 脱敏显示

### 7.3 接口安全
- **限流**: 用户级别、IP级别
- **防刷**: 短信验证码限流
- **SQL注入**: ORM自动防护
- **XSS防护**: 输出转义

## 13. 附录

### 13.1 技术决策记录 (ADR)

#### ADR-001: 选择FastAPI作为Web框架
**状态**: 已接受
**理由**: 高性能、自动文档、类型安全、异步支持
**后果**: 开发效率高，生态不如Django

#### ADR-002: 选择PostgreSQL作为数据库
**状态**: 已接受
**理由**: ACID事务、JSONB支持、全文搜索、稳定可靠
**后果**: 学习成本高于MySQL，但功能更强大

#### ADR-003: 采用微服务架构还是单体架构
**状态**: 已选择单体架构
**理由**: MVP阶段，团队规模小，快速迭代
**后果**: 后期拆分成本高，但当前开发效率高

#### ADR-004: 选择HBuilderX + uni-app作为前端技术栈
**状态**: 已接受
**理由**:
1. 一套代码多端发布（微信小程序、H5、iOS App、Android App）
2. HBuilderX官方IDE，开箱即用，内置调试工具
3. 成熟生态，丰富的插件市场
4. 优秀的性能表现（接近原生）
5. Vue3技术栈，团队易上手
**后果**:
- 正面: 开发效率高，一次开发多端发布，维护成本低
- 负面: 小程序特有API需要条件编译，包体积可能略大
