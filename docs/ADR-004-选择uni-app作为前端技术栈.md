# ADR-004: 选择HBuilderX + uni-app作为前端技术栈

## 状态
已接受

## 背景
需要为云衣橱项目选择合适的前端开发方案，考虑因素包括：
- 开发效率
- 多平台支持（微信小程序、H5、App）
- 团队技术栈熟悉度
- 生态成熟度
- 性能表现

## 决策
选择 **HBuilderX + uni-app (Vue3)** 作为前端技术栈

## 备选方案

### 方案A: 微信小程序原生开发
**优点**:
- 性能最优
- API完整支持
- 官方文档完善

**缺点**:
- 只支持微信小程序
- 无法复用代码到其他平台
- 后期扩展成本高

### 方案B: uni-app (当前选择)
**优点**:
- 一套代码多端发布（微信小程序、H5、iOS、Android）
- HBuilderX官方IDE，开箱即用
- Vue3技术栈，团队易上手
- 优秀的性能表现（接近原生）
- 丰富的插件市场
- 生态活跃，社区支持好

**缺点**:
- 小程序特有API需要条件编译
- 包体积略大于原生
- 部分高级特性需要兼容处理

### 方案C: Taro (京东)
**优点**:
- React技术栈
- 多端支持
- 生态完善

**缺点**:
- 配置相对复杂
- 团队Vue经验更丰富

## 决策理由

### 1. 多端发布能力
云衣橱产品路线规划中包含多端App支持（第三阶段），选择uni-app可以：
- MVP阶段先发布微信小程序
- 使用同一套代码发布H5
- 后期可快速发布iOS和Android App
- 避免重复开发，节省成本

### 2. 开发效率
- HBuilderX官方IDE，开箱即用，无需复杂配置
- 内置调试工具，实时预览
- 丰富的插件市场，开箱即用的组件
- 一键打包发布到各平台

### 3. 技术栈匹配
- Vue3技术栈，团队易上手
- Pinia状态管理，官方推荐
- TypeScript支持良好
- 与现代前端开发习惯一致

### 4. 性能表现
- 接近原生性能
- 优化的渲染机制
- 按需加载，减少包体积
- 良好的用户体验

### 5. 生态成熟度
- DCloud官方长期维护
- 活跃的社区支持
- 丰富的插件和组件
- 持续更新迭代

## 风险评估

### 风险1: 小程序特有API兼容性
**概率**: 中
**影响**: 中
**应对**: 使用条件编译 `#ifdef` 处理平台差异，封装统一API

### 风险2: 包体积过大
**概率**: 低
**影响**: 低
**应对**: 分包加载、按需引入、代码压缩

### 风险3: 性能略低于原生
**概率**: 低
**影响**: 低
**应对**: uni-app持续优化，实际差异可忽略

## 后果

### 正面影响
1. 一次开发，多端发布，大幅降低开发成本
2. 统一代码库，维护成本低
3. 快速迭代，支持敏捷开发
4. 未来扩展性强，可快速支持新平台
5. 丰富的生态，提高开发效率

### 负面影响
1. 小程序特有功能需要条件编译处理
2. 学习曲线需要团队适应Vue3和uni-app
3. 包体积可能略大于原生（但可接受）

## 实施计划

### Phase 1: 微信小程序 (MVP)
- 使用uni-app开发
- 发布微信小程序
- 完成核心功能

### Phase 2: H5网站
- 同一套代码编译H5
- 发布Web版本
- 支持浏览器访问

### Phase 3: 移动端App
- 同一套代码编译iOS和Android
- 发布到App Store和各大应用商店
- 提供原生体验
